## 0. 確定方針（勝彦裁定の写し・本設計の枠）

1. **営業型の協力者はパートナー機構で扱う**（紹介者＋協力タスク＋個別報酬率）。ベンダー＝**納品を伴う受託者のみ**に純化。
2. **ベンダーアプリを3機能に縮小**: ①承諾（1タップ・合意証跡＝本人操作必須）②参加プロジェクトの明細閲覧 ③経費エビデンスのアップロード。納品済み・検収等の**状態操作はMB運営（＋サプライヤー・コンソール）側へ移管**。
3. **売上の裏付けは軽く**: エスクロー型は規制論点につき不採用。実装は (a)任意エビデンス添付 (b)乖離琥珀フラグ の2点のみ。

## 1. 現vendorフロー全接続点と「消える・移る・残る」マップ

現行の状態機械: `proposed →(vendor accept)→ accepted →(vendor deliver)→ delivered →(console 月次freeze)→ delivery_payout_items(unpaid→paid)`。経費: `delivered後 vendor申請(submitted+evidence_path) → console承認(approved)/却下/差戻し → 承認分がpayoutのexpense_totalへ`。

| # | 接続点（ファイル） | 現操作 | 縮小後 | money/検証影響 |
|---|---|---|---|---|
| V1 | `api/vendor/assignments/[id]` action:'accept'/'decline' | 承諾/辞退（vendor） | **残る**（①合意証跡・本人操作必須の核） | なし |
| V2 | 同 action:'deliver' | 納品宣言（vendor） | **移る**→ console案件ドロワー＋サプライヤー案件ドロワーに「納品済みにする」（操作者を`deal_events`に記録） | **なし**＝statusの書き手が変わるだけ。`delivered`ゲート（経費申請可否・payout凍結対象）は不変 |
| V3 | `api/vendor/expenses` POST（evidence_path付き） | 経費申請＋領収書 | **残る**（③の中核）。UI文言を「エビデンスのアップロード」へ寄せる | なし（expense_claims/承認フロー不変） |
| V4 | `api/vendor/expenses/[id]` 取下げ | 申請取下げ | **残る**（③付随） | なし |
| V5 | vendor面 cases一覧/[id]明細（base_fee・経費・状態） | 明細閲覧 | **残る**（②）。PM風の残滓（tasks/updates/deliverables/schedule）は**消える** | なし |
| V6 | `api/vendor/tasks・updates・deliverables・schedule`＋対応UI | 進捗・成果物・予定 | **消える**（405化 or ルート撤去。deliverables のデータは残置=読み取りのみconsoleへ） | なし（moneyに非接続） |
| V7 | vendor面 rewards（支払明細） | 支払明細閲覧 | **残る**（②に含む・delivery_payout_items読み取り） | なし |
| V8 | console 経費承認/却下/差戻し | 運営操作 | **残る**（不変） | なし |
| V9 | console delivery-payouts 月次凍結→paid | 運営操作 | **残る**（不変・byte非接触） | **ゼロ**（凍結ロジック・money検証に一切触れない） |
| V10 | サプライヤー案件ドロワー「委託を提示」（v6） | 提示（supplier） | **残る**＋**追加**: 納品済みにする（V2の移管先。自社案件×自社委託先のみ・audit） | なし |

**結論**: 縮小はUI/権限の再配置のみで、`delivery_payout_items`・経費承認・money検証（3+1ハッシュ）への影響は**ゼロ**。検収という独立状態は導入しない（`delivered`のままconsole/supplierが宣言者になる＝状態機械不変が最小リスク）。

## 2. オムニス委託先（営業型）の移行設計

- **実データ実査**: supplier所有deliveriesの実アサインは現在0件（v6は検証のみ）。**移行すべき既存データは存在しない**——本件は「これから間違った席に座る人を防ぐ」設計である。
- **機構の対応表**: 営業型がベンダーで持っていた性質→パートナー機構での受け皿：
  - 実務・営業の分担 → **協力タスク**（メニュー別・reward_id紐付け・既存）
  - 個別の取り分 → **パートナー別報酬率**（partner_reward_overrides・既存）
  - 網への帰属 → frontier_id（サプライヤー配下・既存）
- **手順（実装バッチP1）**: ①サプライヤー「委託先を招待」モーダルに分岐質問を追加——「この方は営業・紹介もしますか？」→はい=パートナー招待（?f=自分）へ誘導／いいえ=従来の委託先招待。②コンソールのサプライヤー詳細・委託先一覧に「営業型はパートナーへ」の1行ガイド（ⓘ）。③既存データ移行ツールは**作らない**（対象0件。将来発生時は個別対応が安全）。
- **高さんへの説明（1段落・そのまま使える文面）**:
  > 「営業やご紹介を担う方は、今後は『パートナー』としてお迎えください（招待はパートナー画面から）。紹介の成果がそのまま報酬になり、メニューごとの役割分担（協力タスク）や、その方だけの特別な報酬率も設定できます。『委託先』は、保険の事務手続きや納品など、実務を担う方専用の席になります——委託費のご提示と、完了・経費の管理はこれまで通りです。」

## 3. 売上エビデンス（確定方針3.の設計）

**重い検証機構を作らない理由（明記）**: 受注額はパートナー報酬とMB手数料の**共通ベース**である。サプライヤーが受注額を過少申告すると、紹介したパートナーへの報酬も過少になり、**パートナー本人のアプリに（自分の紹介案件の成約として）可視化される**。つまり過少申告は自社の紹介網への背信としてただちに露見する自浄構造が既に働いており、システム的な突合装置は費用対効果が合わない。**エスクロー型（MB代金受領→差引→振込）は不採用**——資金移動業（資金決済法）該当リスク、保険商品では保険料収納代行の保険業法論点があり、ライセンス無しに踏み込まない。

### (a) エビデンス添付欄（任意）
- **ストレージ**: 既存の`expense-evidence`バケットと同流儀で新バケット`deal-evidence`（private・signedUrl 60s）。実装は`api/vendor/expenses`のアップロード/参照コードを流用。
- **データ**: `deal_evidences`テーブル（additive: id/deal_id/uploaded_by_partner_id/path/label/created_at）。dealへの列追加ではなくN件対応（契約書＋請求書）。
- **入口**: サプライヤー案件ドロワーの受注額入力の直下に「エビデンスを添付（任意）」——**任意ポリシー明記**: 「通常は不要です。大型案件やお取引の記録を残したいときにご利用ください」（静音・義務化しない）。
- **参照導線**: コンソール案件ドロワー（受注額行の隣に📎アイコン）＋サプライヤー請求画面の該当行（署名URL・owner/managerのみ）。
- **面公開禁止との整合**: エビデンスは供給者→MBの文書。**一般パートナー面・受託者面には一切出さない**（APIはsupplier本人=自案件のみ／console=owner/manager。/api/servicesや共有面には非掲載）。

### (b) 乖離琥珀フラグ（静音・入力ミス検出兼用）
- **判定基準案**: 同一メニューの確定済み受注額（直近90日・N≥3）の**中央値から±70%超の乖離**、またはN<3のとき**メニュー報酬設定から逆算した想定受注額**（fixed報酬額×10倍等の粗い帯）から1桁ずれ。閾値は`lib/supplier-fee.ts`に定数（調整可能）で置き、**保存はブロックしない**（フラグのみ＝静音）。
- **表示位置**: ①コンソール案件ドロワーの受注額行に琥珀ドット＋「過去平均から乖離（¥X中央値）」1行 ②サプライヤー請求のクローズプレビュー行に同フラグ（凍結前に気づける位置）③サプライヤー側には出さない（本人への嫌疑表示は関係を損なう。入力直後の桁ミスのみ本人トースト「入力値をご確認ください（¥X,XXX,XXX）」で優しく確認）。
- **監視接続**: /api/monitor tier3に「未解消の乖離フラグn件」を追加可能（P2・任意）。

## 4. 請求書発行代行（Phase 2骨子・実装しない）

MBは既にサプライヤー請求で「金額の確定と履歴」を持つ。Phase 2では①supplier_charges確定時に**適格請求書PDF（インボイス番号・税区分）を自動生成**（既存brandedメール基盤＋PDF生成をVercel内で・外部SaaS不要）②サプライヤーポータル「お金」から自社宛請求書DL③さらに先: サプライヤー→パートナー方向の**支払明細書の代行発行**（MBが支払データを持つため自然な拡張）。法的位置づけは「発行事務の代行」（代金収納はしない＝エスクロー回避と同じ線引き）。前提: partners.company_name（v9済）・インボイス登録番号（partners.invoice_number 既存列）・消費税表示の税務レビュー1回。

## 5. リスク・バッチ分割・規模

| リスク | 手当て |
|---|---|
| vendor納品宣言の移管で「納品を知る人＝受託者」の自然さを失う | consoleに加えサプライヤー（発注者）にも宣言権を置く（V10）。deal_eventsに宣言者を常時記録し争議に備える |
| V6撤去でvendor面の既存導線が404 | ルートは残しリダイレクト（案件詳細へ）・撤去はUI導線のみ |
| 乖離フラグの誤検知（新メニュー・単発大型） | ブロックしない設計＋N<3は緩い帯のみ＋コンソールで「確認済み」にできるdismiss |
| 添付ファイルの個人情報 | privateバケット・署名URL60s・console閲覧をaudit記録 |
| 営業型分岐質問の無視（全部委託先に入れる） | 招待モーダルの分岐を既定「営業もする＝パートナー」に倒す＋コンソールⓘ |

**バッチ分割・見積り**: **P1**（1バッチ・実装1日相当）=V2移管（console/supplierの納品ボタン＋vendor側deliver 405化）・V6縮小（PM残滓の導線撤去）・vendor面3機能純化の文言/ⓘ・招待分岐質問。**P2**（1バッチ）=deal_evidences添付＋参照導線＋乖離琥珀フラグ。**P3**（別卓）=請求書発行代行。E2E拡張は各+8〜12 assertions（vendor受諾はモック不可＝実ブラウザ）。
