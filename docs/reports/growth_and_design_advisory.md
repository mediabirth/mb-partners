## 0. 最重要の発見（Claudeと勝彦に見えていない事実）

**「11ギャップ」の多くは、既に 60〜90% 実装済みです。** 未実装ではなく**未活性化**。データモデルとルートを実測した結果:

## 1. リストへの批評と追加

### Claude案（3+4 → 9 → 1 → 10）への評価
- **方向は同意**（紹介の入口と循環が核心）。ただし **3/4/9/10 は既存資産の活性化**であり「新規実装」の見積りは過大。
- **順序に一部反対**: 10（ファネル計測）を最後に置くのは危険。3を露出した瞬間に**計測が無いと効果が測れない**。3の露出と10の集計は**同時に点火**すべき（funnel_events は既に記録土台がある）。

### 見えていない機会・追加項目
- **A. 共有UIの露出（最優先の隠れ資産）**: /r/[token] と referral_links が既にあるのに、パートナーが自分のリンクを取得・共有する画面が無い。**1コミット級で紹介の入口が開通**する（後述クイックウィン）。
- **B. funnel_events の発火強化＋集計**: view/submit/成約の3点で dedup_hash 付き記録は設計済み。あとは**発火の網羅**と**転換率ダッシュボード**（console）。データ構造は対応済み＝安い。
- **C. 「思い出す」トリガーの自動化**: reminder-* が既にあるので、休眠パートナーへの**自動ナッジ（最終紹介からN日）**は cron＋既存メール雛形で安く出せる。新メニュー告知は broadcasts を**サービス公開イベントに自動連動**させるだけ。
- **D. 社会的証明（task2）は「重い」側**: 実績フィードは匿名化・鮮度・不変条件（顧客名/金額の露出禁止）との干渉があり、見た目より重い。優先度は下げるべき。
- **E. 落とし穴の警告**: 顧客セルフ登録（4）は**同意記録の法的証跡**（consent + timestamp + IP/文言バージョン）が要る。/r/[token] は consent を取るが、**同意文言のバージョン固定と監査保存**を確認・強化すべき（identity/money 不変条件とは無干渉だが法務証跡は別軸）。

## 3. 最優先1〜2件の実装スケッチ

### ① 共有UI露出＋ファネル点火（推奨・最初の1バッチ）
- **データモデル**: 追加ゼロ。`referral_links`（既存）・`funnel_events`（既存）を使う。
- **面ごとの画面**:
  - パートナーAPP `/app/refer`（またはサービス選択後）に「**紹介リンクを共有**」ブロックを追加＝`getOrCreateReferralLink(serviceId)` を呼び token 取得 → `https://mb-partners.app/r/{token}` を**コピー / LINEで送る / QR**。
  - 既存 `/r/[token]` はそのまま入口（改修不要）。`trackFunnel('view')` は組込済、`submit`/`won` の発火を網羅。
  - コンソール `/console` に**ファネル集計**（funnel_events を token→partner で集計：閲覧→送信→成約の転換率・パートナー別生産性・最終活動日）。
- **外部依存**: LINE共有は `https://line.me/R/msg/text/?{encodeURIComponent(text+url)}`（LIFF不要・URLスキームのみ）。QRは既存 `qrcode` 依存で生成可。
- **落とし穴（既確立不変条件との干渉）**:
  - **money**: 触れない（紹介リンクは token 帰属のみ・報酬計算は成約時の既存ロジック）。★menu_rewards 16/340,100・報酬ハッシュ不変を維持。
  - **identity/セッション**: /r/[token] は**未認証公開ページ**＝ auth cookie を書かない（session分離の中央factoryを経由しない＝OK）。顧客submitは deal 起票のみ・profiles/partners 非接触＝identity不変条件と無干渉。
  - **外部送信ガード**: 共有はユーザー操作（コピー/LINE）＝サーバからの自動送信ではない。funnel記録は内部書込のみ。
  - **RLS**: referral_links/funnel_events の RLS を確認（本人partnerのlinkのみ生成・funnelはservice_role書込）。
- **概算規模**: 共有UI（1画面ブロック）＋LINE/QR＋funnel集計console＝**中1バッチ**（実装コミット2〜3・DDLゼロ）。

### ② LINE共有intent（①に内包 or 単独クイックウィン）
- URL生成のみ。`line.me/R/msg/text/?` に「{パートナー名}さんからのご紹介です {url}」。コスト最小・効果大（日本の共有主動線）。

## 5. 反対意見（開発者として遠慮なく）

- **2（社会的証明フィード）を今やるな**: 効果はあるが、匿名化・鮮度・顧客名/金額の露出禁止（確立済み不変条件）との干渉で**見た目より重く、事故ると信頼を損なう**。紹介入口（A）が回り始め、実データが貯まってからで十分。
- **「11個を作る」という枠組み自体に反対**: 実測では大半が実装済み。**新規構築のロードマップではなく、既存資産の活性化リスト**として再定義すべき。工数見積りが一桁変わる。
- **計測（10）を後回しにするな**: 入口を開ける前に集計の器を用意しないと、施策の可否が判断できない。3と10は不可分。
- **メニュー棚（11）は開発課題ではない**: サービス/メニューを足す運用。開発優先度から外すべき。

## 6. デザイン・アイデンティティ刷新（v3）の技術見解

### (a) 全面視覚刷新の安全性・コスト — **色は安い、書体は要注意**
- **トークン化率 ~94%**: 実測で `var(--blue)/--c-blue/--txt/--line/--muted` 等の**トークン参照 1,560箇所** に対し、**インディゴ直書き(#4733E6等) 105箇所**（約15ファイル）。
- **脱インディゴの配色刷新** = `app/globals.css` の `:root`（`--c-blue` 系・`--st-*` 意味色）を差し替えれば**3面すべてが追随**（トークン経由の1,560箇所は自動反映）。残る**手作業は105箇所**——その多くはロゴSVG・ヒーローのグラデーション（#5240F2→#3D2BD0）・splash・manifest theme_color。**中1バッチで色ピボット完了可能**。
- **radius/影/ease はトークン化済**（`--radius*`・`--shadow*`・`--ease-*`）＝一括差替可。ただし**inline boxShadow が41ファイル**に残存＝影の署名変更は中規模の掃除を伴う。
- **署名的コンポーネント・モーション署名**は新規デザイン作業（トークンでは吸収できない造形）＝ここが v3 の本体コスト。トークン刷新は安いが「AIっぽさの脱却」はコンポーネント造形とタイポ次第。

### (b) 日本語Webフォント — **ネイティブ体感目標との最大の緊張点**
- 現状: `next/font` で Inter（latin subset）＋ Zen Kaku Gothic New（**subsets:['latin'] のみ**）。**日本語グリフは subset 不可＝フル woff2 を配信**（数百KB〜数MB）。既に日本語表示はこのフルフォント依存。
- **v3で重い日本語書体に替えると、コールドスタート（ネイティブ体感プログラムで既に束律速・app 30s/console 10s＠4x）がさらに悪化**する。これが最大の落とし穴。
- **最適戦略**:
  1. **サブセット化**: 実際に使う日本語グリフだけを抽出した woff2 を**自前ホスト＋`unicode-range`＋`font-display:swap`**（ビルド時 glyph 抽出）。next/font の JP フル配信より桁で軽い。
  2. **表示用と本文用を分離**: 見出し（display）だけ署名書体（少数グリフ＝サブセット効果大）、本文は system-ui/軽量書体。
  3. **preload は見出し1ウェイトのみ**。多ウェイト同時読込を避ける。
  4. **`size-adjust`/フォールバック整合**でスワップ時のレイアウトシフトを抑制。
- 結論: **色ピボットはperf無害で即可。書体刷新はサブセット戦略必須＝束削減プログラムと同じ土俵**。

### (c) 3面一括刷新の推奨手順とリスク
1. **トークン層で完結する部分（色・radius・影・ease）を先に**：globals.css の :root 差替＋105色/41影の掃除→**3面自動追随**を対比スクショで検証。リスク低（意味色 --st-* を壊すとステータス表現が崩れるので、意味色マップは慎重に）。
2. **署名コンポーネント/タイポは面ごとに段階適用**（APP→vendor→console）。各段でparity証跡（前バッチで確立した1:1対比）を取る。
3. **リスク**: (i) 意味色（success/warn/danger）の可読性・アクセシビリティ、(ii) ハードコード見落としによる「半分だけ新色」、(iii) 書体スワップのCLS/perf。いずれも**トークン集中と対比スクショ**で管理可能。

### (d) Growth と デザイン刷新の実施順序 — **Growthを先に、色は並行可、書体は後**
- **Growth（共有UI＋計測）を最優先**: 本番が始まり事業の核心は紹介。**入口開通は今すぐ効く**。デザイン刷新は情緒価値だが紹介の増加に直結しない。
- **色ピボットは Growth と並行可能**（perf無害・トークン層・機能非干渉）。むしろ Growth の共有ページ（/r/[token]）を新色で出せば対外ブランド刷新の初手になる。
- **書体刷新は後（束削減の後）**: (b)(7)の理由で、書体はビルド/資産に重く効く。

## 7. 束削減プログラムと v3 の先後（追補2への回答）

- **束削減（/console/deals分割・app home RSC移譲）と v3書体刷新は同じ土俵**（両方ビルド/資産構成＝webpack chunk・font asset・critical CSS を触る）。**同時に着手すると回帰の切り分けが困難**（perf悪化が束かフォントか分からない）。
- **推奨順序**:
  1. **色ピボット（トークンのみ）** … いつでも・Growthと並行OK・perf無害。
  2. **束削減プログラム** … コールドスタートの土台を軽くする（計測付き・単独で before/after）。
  3. **書体刷新（サブセット戦略）** … 軽くなった土台の上に載せる。フォント追加の perf コストが**軽い基盤**に着地するので安全。
- **統合は非推奨**: 束削減とフォントを1バッチにすると、ネイティブ体感の数値目標（cold<2s）に対する各施策の寄与が不可分になり、未達時の原因究明ができない。**別プログラム・別計測**で。

## 総括（開発者としての結論）

1. **紹介ループの配管は既に敷設済み**。最大のROIは新規開発ではなく **/r/[token] 共有UIの露出＋ファネル点火**（クイックウィン級）。Claudeの方向性は正しいが「作る」を「通水する」に読み替えよ。
2. **順序**: A(3+4+10同時) → 5(LINE) → C(自動トリガー) → 9 → 6 → 7。2(社会的証明)は後回し、11は運用。
3. **デザインv3**: 色は安い（トークン94%）・書体は束削減の後。**Growth優先、色は並行、束削減→書体の順**。
4. **不変条件**: どの施策も money（16/340,100・ハッシュ）・identity（role/name非上書き）・session分離・外部送信ガードと**構造的に無干渉**に設計可能（/r/[token]は未認証公開＝auth非接触が鍵）。
