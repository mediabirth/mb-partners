# MB Partners 文言規範（Copy Guideline）v1 — 2026-07-04

3面（パートナーAPP・管理コンソール・ベンダーポータル）＋メール＋公開ページ（join/book/legal）の全UI文言の単一規範。
デザインシステムv2.2（静かな規律）の「言葉」版。新規文言は必ず本規範に従う。

## 1. 用語辞書（単一の正）

| 正 | 禁止・誤 | 備考 |
|---|---|---|
| お客さま | お客様・顧客 | 敬称は lib/customer.ts 経由（個人=氏名 様／法人=法人名 様）。地の文・ラベル・placeholder も「お客さま」 |
| 案件 | ディール・deal | |
| 紹介 | リファラル（UI本文で） | 種別名としての「リファラル/フロンティア」チップは可 |
| 報酬 | リワード | パートナー向け。委託先向けは「委託費」 |
| 成約 | 成約・確定 | lib/status.ts が単一ソース |
| 支払済 | 支払済み | |
| 承認済／却下 | 承認済み・却下済み | 名詞形で統一 |
| 未払い | 未払 | |
| ご契約（お客さま宛） / 成約（パートナー宛） | — | 同じ事象でも相手により語を変える（公式） |

**ステータスは4語＋終端**: 受付／対応中／成約／支払済（進行）＋不成立（終端）。この5語以外の状態語をパートナー面に出さない。ベンダー面は lib/vendor-status.ts のベンダー語（受付/実行中/完了/終了）を公式の別表とする（報酬・成約の非開示設計のため意図的に別語）。

## 2. 表記・記号

- **括弧は全角（）**。税表記も「（税抜）」「粗利（税抜）」。半角()は英数式・コード表記のみ
- **区切り記号は中黒「・」**。インタープンクト「·」(U+00B7)は禁止。罫線「─」は「サービス ─ メニュー」の構造対のみ許可
- **三点リーダは「…」**（半角...禁止）。疑問符は全角「？」。**感嘆符「！」はUI禁止**（メール含む）
- 例示placeholderは「例：」（全角コロン）。「例)」「例）」「例: 」禁止
- 任意/必須は「（任意）」「（必須）」。「（省略可）」禁止
- 金額は「¥」＋3桁区切り。単位は 件・名・月
- 日時はJST明示（timeZone:'Asia/Tokyo'）・年は文脈上自明なら省略可

## 3. ボタン

- 実行・確定・破壊は**動詞で終える**: 保存する／送信する／削除する／承認する／招待する
- 離脱・中立は固定語: キャンセル／閉じる／戻る
- 例外: コンソールの高密度な表内アクション（行内の 承認・却下・削除 等）は名詞形可（明文化された唯一の例外）
- 記号付きボタン禁止（「✓」「→」等）。ナビの「›」「←」は可

## 4. エラー・トースト・空状態

- 型: **「〈対象〉を〈動作〉できませんでした。〈次の行動〉」**。例:「保存できませんでした。時間をおいて再度お試しください」
- リトライ文言は「時間をおいて再度お試しください」に一本化（「再試行」「再読み込みしてください」禁止）
- 汎用「エラーが発生しました」禁止＝必ず対象を明示
- 依頼は「〜を入力してください」（「ご入力ください」等の過剰敬語は使わない。ただし顧客向け予約/メールの「ご連絡」「ご確認」等の名詞敬語は可）
- トースト・インラインエラーは**末尾句点なし**。複数文になる場合のみ文間に句点
- 空状態は「〈何が〉はまだありません」＋（必要なら）次の一歩を1文。煽らない

## 5. トーン

- **APP（パートナー面）**: 静かな敬体。絵文字・感嘆符・煽り（「さあ〜しよう」「知っていますか？」）禁止。称賛は事実＋感謝で表す
- **メール**: 件名「【MB Partners】＋受動・完了トーン」（〜しました／〜のご案内）。署名「— MB Partners 運営事務局」。本文は宛名→リード1文→明細→次の案内
- **コンソール**: 簡潔・中立。表内は名詞形可。絵文字は使わない（機能アイコンはSVG）
- **ベンダー**: APPと同じ静かな敬体＋委託費言語
- 冗長禁止: 自明な括弧補足（「（承認済）」等）・「〜することができます」・二重敬語

## 5b. コンソール静音化v2（恒久原則・2026-07-04追加）

1. **言葉は操作の瞬間にだけ現れる** — 確認ダイアログ・トースト内のみ。常設の説明文/ヒント文/結果予告テキストは禁止
2. **ラベルが結果を語る動詞ボタン**（例「連絡済みにして商談中へ」）で完結させる
3. **1画面1焦点** — 開いた瞬間の視線の落ち先は1つ
4. **等圧カードの入れ子禁止** — 枠の中に枠を作らない。区切りは余白と0.5px罫線のみ
5. **需要時表示** — 写像=ホバー/ⓘ、金額原価=フェーズ連動。構造（配置・順序・余白・ラベル）が進め方を語る

## 5c. 内部語彙の対外露出禁止＋対外呼称正典（恒久原則・2026-07-05追加）

- **禁止**: ユーザー可視文字列（見出し・ラベル・プレースホルダ・バッジ・ボタン・aria-label・PWA名/説明・メール本文/件名・通知文）に、内部役割語彙「デリバリー」「ベンダー」「vendor」「delivery」を出さない。これらは**コード（パス・変数・コメント・DBキー）専用**。
- **対外呼称の正典**:
  - 面の名称・ヘッダ・PWA名 = **「MB Partners」のみ**（役割語を付けない）。
  - パートナー面のユーザー = 「パートナー」（対外ブランド語）。委託先面のユーザーには対外役割語を用いず、**氏名/屋号＋ID表記**で示す。
  - **ID表記**は3面同型: `chip chip-referral`・Inter・letterSpacing .08em。パートナー=partners.code、委託先=deliveries.display_code（英字2+数字4・同採番）。
- **例外（法務・契約文のみ）**: 業務委託規約・契約文書は契約用語「受託者」「業務委託先」を使用可。ただし「デリバリー」「ベンダー」は使わない。
- **検証**: `grep -rn "デリバリー\|ベンダー" app/vendor lib`（コメント除く）でユーザー可視露出ゼロを確認する。

## 5d. 存在しない機能の文法を移植しない（恒久原則・2026-07-05追加）

- parity（3面整合）は**構造・文法の一致**であって、**存在しない機能のUIを空で移植すること**ではない。
- ある面に実体（データソース・配信機構・運用フロー）が無い機能のタブ・カード・入力は**置かない**。空タブ/常時ゼロ表示は「更新されない第二の真実」＝張りぼて予備軍。
- 例: 受託者面に「お知らせ」タブは置かない（broadcasts に相当する受託者向け配信機能が無いため）。通知は本人宛イベントの単一リストにする。
- 判定: 「この面にこの機能の実体はあるか？」が No なら、parity を理由に移植しない。実体が生まれた時点で追加する。

## 5e. 顧客向け公開ページの文言規律・正式社名（恒久原則・2026-07-06追加）

- **正式社名の正典** = **「株式会社Media Birth」**（前株）。フッター・同意文・法務・対外表記はすべてこの表記に統一する（「Media Birth株式会社」＝後株や「メディアバース」は不可）。文中で会社を指す口語は「Media Birth」可（例：完了画面「Media Birth の担当者よりご連絡いたします」）。
- **対外禁止語に「MB」（略語）を追加**。顧客向け公開ページ（`/r/[token]`・`/book`・`/legal` 等、被紹介の顧客が見る面）のユーザー可視文字列に略語「MB」を出さない。運営主体は「株式会社Media Birth」／口語「Media Birth」で表す。※パートナー面/コンソール等のプロダクト名「MB Partners」は 5c のとおり従来どおり（顧客向け公開ページには持ち込まない）。
- **顧客向け公開ページの新規文言は、データ正典または勝彦/Claude 承認済み文言に限る（CC の言い換え・生成を禁止）**。見出しに使うサービス/メニュー名は必ず正典データ（`services.name`・新 `menus.name`）由来とし、非選択メニュー名を主役化しない（リンク粒度＝パートナーの明示選択のみが主役）。「無料」「費用」等の費用言及、「成約」「報酬」「金額」等の金銭語彙は顧客向け公開ページに置かない。
- **検証**: `grep -nE "無料|費用|成約|報酬|金額|MB[^-]|デリバリー|ベンダー" app/r/\[token\]/page.tsx`（コメント除く）でユーザー可視露出ゼロを確認する。

## 5f. サプライヤー／フロンティアの呼称正典（恒久原則・2026-07-12追加）

- **サプライヤー＝法人・メニュー供給元**（`partners.supplier_rate_card` またはブランド結線 `services.supplier_partner_id` を持つ会社）。コンソール全域の表示区分・バッジ・フィルタは「サプライヤー」で統一し、「フロンティア」「会社フロンティア」表記と混在させない。
- **フロンティア＝個人の紹介網主**（12ヶ月ウィンドウのoverride対象）。サプライヤーは機構上 is_frontier=true を共有するが、**表示層では常に区別**する（機構・データは共通のまま）。
- レートカード呼称: 標準＝**standard-v2（パススルー＋受注額5%）**＝系統外はパートナー報酬をパススルー支払いし、MB手数料=受注額（税抜）5%を別建てでサプライヤーに請求。**折半（粗利50%）はオムニス等の個別契約カード専用**の形。粗利ベースの語は標準カードの文脈で使わない。
- 標準サプライヤーのメニュー報酬型は「固定額」または「受注額%」のみ（粗利%・継続は個別契約カードのみ）。
- **面公開禁止の改定（2026-07-12・サプライヤーポータル）**: fee/レートカード/請求情報は一般パートナー面・受託者面に不出（不変）。ただし**サプライヤー本人には自社分のみ表示可**（本人が支払う当事者のため）。対象は自社の供給ブランド・案件（受注額まで。他パートナーの報酬額は不可）・自社宛請求・自社案件の委託サマリに限る。スコープは必ずセッション由来の partner id で強制し、URL/クエリからサプライヤーidを受けない。
- 請求状態の対外語彙（サプライヤー向け）: unbilled=「締め済み・請求書待ち」／invoiced=「請求済み」／settled=「お支払い確認済み」。内部語（unbilled等）を画面に出さない。
